# Context Mode for Codex CLI Design

## Goal

Install and configure `context-mode` for the user's Codex CLI setup while
preserving the existing dotfiles structure and Codex instruction files.

The selected approach follows the upstream Codex CLI instructions closely:
register the MCP server in Codex config, put context-mode hook registration in
`$CODEX_HOME/hooks.json`, and do not add context-mode routing instructions to
the global Codex `AGENTS.md`.

## Current State

The dotfiles repository manages Codex settings under `codex/.codex/`.
Durable checked-in settings live in `codex/.codex/config.base.toml`.
`codex/.codex/config.toml` is generated by `bin/.local/bin/codex-sync` and is
intentionally gitignored because Codex writes local runtime state there.

Codex hooks are already enabled in `config.base.toml`:

```toml
[features]
hooks = true
```

The existing atomic commit guard is registered as a TOML `PreToolUse` hook.
That hook and `codex/.codex/AGENTS.md` should remain unchanged.

## Requirements

- Install `context-mode` globally with npm.
- Add the `context-mode` MCP server to the durable Codex base config.
- Add a tracked `codex/.codex/hooks.json` containing the upstream Codex
  context-mode hook registrations.
- Preserve the existing atomic-commit hook registration.
- Do not modify `codex/.codex/AGENTS.md`.
- Regenerate the local Codex config from `config.base.toml`.
- Verify that the package is installed, the MCP server config is present, and
  the hooks file is valid JSON.

## Design

### Package Installation

Install the package globally:

```bash
npm install -g context-mode
```

This makes the `context-mode` executable available to Codex for both the MCP
server command and hook commands.

### Codex MCP Configuration

Add the MCP server to `codex/.codex/config.base.toml`:

```toml
[mcp_servers.context-mode]
command = "context-mode"
```

The existing `[features].hooks = true` remains in place. The existing TOML
atomic-commit hook remains in place.

After editing the base config, regenerate `codex/.codex/config.toml` with:

```bash
DOTFILES=/home/benkim0414/workspace/dotfiles/.worktrees/context-mode-codex \
  bin/.local/bin/codex-sync
```

When applying from the primary checkout, the normal `codex-sync` invocation is
enough because `DOTFILES` defaults to `$HOME/workspace/dotfiles`.

### Codex Hook Configuration

Create `codex/.codex/hooks.json` with the upstream Codex hook shape:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "local_shell|shell|shell_command|exec_command|Bash|Shell|apply_patch|Edit|Write|grep_files|ctx_execute|ctx_execute_file|ctx_batch_execute|ctx_fetch_and_index|ctx_search|ctx_index|mcp__",
        "hooks": [
          {
            "type": "command",
            "command": "context-mode hook codex pretooluse"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "context-mode hook codex posttooluse"
          }
        ]
      }
    ],
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "context-mode hook codex sessionstart"
          }
        ]
      }
    ],
    "PreCompact": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "context-mode hook codex precompact"
          }
        ]
      }
    ],
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "context-mode hook codex userpromptsubmit"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "context-mode hook codex stop"
          }
        ]
      }
    ]
  }
}
```

This file is tracked because it is durable configuration for the stowed
`~/.codex` package. Codex should see it as `$CODEX_HOME/hooks.json` when the
dotfiles package is stowed.

## Data Flow

1. Codex starts and reads `~/.codex/config.toml`.
2. The MCP server entry launches `context-mode` as an MCP server.
3. Codex hooks are enabled by `[features].hooks = true`.
4. Codex reads `~/.codex/hooks.json`.
5. Context-mode receives hook events for tool use, session start, compaction,
   user prompts, and stop events.
6. Existing atomic-commit enforcement continues through the current TOML
   `PreToolUse` hook.

## Error Handling

- If `npm install -g context-mode` fails because of network or permissions, rerun
  the command with the required approval or report the package-manager error.
- If `context-mode doctor` reports missing runtime support, follow the doctor
  output before declaring the configuration complete.
- If Codex does not pick up `hooks.json`, verify the stowed path is
  `~/.codex/hooks.json` and that `[features].hooks = true` exists in the
  generated config.
- If both the TOML atomic hook and `hooks.json` context-mode hook fire on
  `PreToolUse`, preserve both; they enforce different policies.

## Testing

- Run the existing atomic-commit hook test before and after changes:

```bash
bash codex/.codex/tests/test-atomic-commits-hook.sh
```

- Validate the new hook file:

```bash
jq empty codex/.codex/hooks.json
```

- Regenerate config in the worktree:

```bash
DOTFILES=/home/benkim0414/workspace/dotfiles/.worktrees/context-mode-codex \
  bin/.local/bin/codex-sync
```

- Confirm the generated config includes the MCP server:

```bash
rg -n "mcp_servers.context-mode|command = \"context-mode\"" codex/.codex/config.toml
```

- Verify the installed package:

```bash
context-mode doctor
```

## Out of Scope

- No changes to `codex/.codex/AGENTS.md`.
- No project-local routing instruction file.
- No migration of the existing atomic-commit hook from TOML to `hooks.json`.
- No changes to Claude, Gemini, Cursor, or other agent configurations.
