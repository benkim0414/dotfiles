# Fix Claude Code settings.json permission warnings

Date: 2026-06-12
Status: Approved

## Problem

On `claude` launch, the harness prints permission warnings:

```
Permission deny rule "MultiEdit(**/.env)" matches no known tool — check for typos.
... (10 more MultiEdit deny rules)

Settings Warning /Users/ben/.claude/settings.json
  permissions.allow: Invalid permission rule "mcp__*" was skipped:
  Wildcard tool name "mcp__*" is not supported in allow rules.
```

Two root causes:

1. **`MultiEdit` removed from Claude Code.** The tool was folded into `Edit`.
   Every `MultiEdit(...)` rule now references a non-existent tool. The harness
   warns for each deny rule and silently skips them.
2. **Bare `mcp__*` invalid in `allow`.** Allow patterns must name the scope they
   widen: `mcp__<server>__*`. Bare wildcards are permitted only in `deny` and
   `ask` rules. The `allow` entry is skipped entirely.

`settings.json` is generated by `claude-sync` from
`claude/.claude/settings.base.json` (+ optional work overlay). Fixes go in the
**base** file, never the generated `settings.json`.

## Changes

All edits to `claude/.claude/settings.base.json`.

### 1. `permissions.allow`

- Remove `"MultiEdit"` (removed tool).
- Remove `"mcp__*"` (invalid in allow).

Per the chosen approach, MCP tools are no longer pre-approved. Under
`defaultMode: "auto"`, the classifier judges each MCP call. This trades zero
maintenance for occasional classifier prompts. Existing `ask` mutation rules
still gate writes.

### 2. `permissions.deny`

Remove all 11 `MultiEdit(...)` rules:

```
MultiEdit(**/.env)
MultiEdit(**/.env.*)
MultiEdit(~/.ssh/*)
MultiEdit(~/.gnupg/*)
MultiEdit(~/.aws/credentials)
MultiEdit(~/.claude/.credentials.json)
MultiEdit(~/.kube/config)
MultiEdit(~/.docker/config.json)
MultiEdit(~/.netrc)
MultiEdit(~/.config/gh/hosts.yml)
```

(Plus the `MultiEdit(**/.env.*)` variant.) Each has a paired `Edit(...)` rule
covering the identical path, so secret-path protection is unchanged.

### 3. Hook matchers

Drop the `MultiEdit` token from 3 matchers (dead reference to removed tool):

- `worktree-guard.sh`: `Write|Edit|MultiEdit|NotebookEdit` -> `Write|Edit|NotebookEdit`
- `permission-policy.sh`: `Bash|Write|Edit|MultiEdit|NotebookEdit|WebFetch` -> `Bash|Write|Edit|NotebookEdit|WebFetch`
- `audit-log.sh`: `Bash|Write|Edit|MultiEdit|NotebookEdit|CronCreate|...` -> same without `MultiEdit`

### 4. `permissions.ask` — unchanged

- `mcp__*__*create*` and the other 11 mutation globs stay valid (deny/ask accept
  bare wildcards) and still gate MCP mutations under auto mode.
- `mcp__plugin_context-mode_context-mode__ctx_purge` and `ctx_upgrade` stay.

### 5. Doc — `CLAUDE.md` "Permission posture"

Rewrite the bullet describing `permissions.allow: ["mcp__*", ...]`. New posture:

- MCP calls are not pre-approved; the auto-mode classifier judges them.
- `ask` rules still gate MCP mutations and the two destructive context-mode tools.
- Drop the "ask overrides allow per precedence" framing — allow no longer lists MCP.

## Verification

1. Run `claude-sync` to regenerate `~/.claude/settings.json`.
2. Confirm the generated file contains no `MultiEdit` references and no `mcp__*`
   in `allow`.
3. Launch `claude` (or equivalent settings load) — confirm zero
   "matches no known tool" warnings and no "mcp__* was skipped" warning.

## Non-goals

- No change to secret-path protection (Edit/Write/Read/Bash deny rules intact).
- No change to the work overlay or `claude-sync` merge logic.
- No enumeration of per-server MCP allow rules (explicitly rejected in favor of
  auto-mode classification).
